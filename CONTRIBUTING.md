# Contributing to Sur9e

Thanks for considering a contribution! Sur9e is a free, self-hosted OSS
project built at side-project pace — focused PRs and well-described issues
are the most valuable thing you can send.

## Before you start

- Read [`CLAUDE.md`](CLAUDE.md) / [`AGENTS.md`](AGENTS.md) (same content,
  different agent runtimes). They are the operating manual: architecture,
  critical rules, and the patterns your change must follow.
- Check existing [GitHub Issues](https://github.com/arspesk/sur9e/issues)
  before opening a new one or starting a large change. For anything bigger
  than a bugfix, open an issue first so we can agree on the approach.

## Dev setup

```bash
git clone https://github.com/arspesk/sur9e
cd sur9e
npm run setup        # guided wizard — deps, Playwright, Python venv, CLI + settings
npm run doctor       # verifies your environment end to end
npm run web          # dev server → http://localhost:3000
```

You don't need real personal data to develop — copy the templates from
`content/examples/personalization/` into `inputs/personalization/`.

## Quality gates

Every commit runs the same gate three times: a post-edit lint hook (when
working with an AI agent), a pre-commit hook, and CI.

```bash
npm run lint         # Biome (TS/JS/CSS/JSON) + Prettier (MD/YAML)
npm run typecheck    # tsc over src/**
npm run test:unit    # vitest
npm run test:e2e     # Playwright (needs the dev server running)
npm run test:quick   # the full pre-commit/CI gate
```

A PR that fails `npm run test:quick` will not be merged. New behavior needs
tests. Behavior-preserving refactors must leave existing tests passing
unmodified.

## AI-assisted contributions

Sur9e is itself built largely with AI coding agents, and AI-assisted PRs are
welcome — under these rules:

1. **You own what you submit.** Review and understand every generated line.
   "The agent wrote it" is not a review.
2. **Agents follow the manual too.** Point your agent at `CLAUDE.md` /
   `AGENTS.md`; the critical rules there (no auto-submit, server logic in
   `src/lib/server/`, tokens in `tokens.css`, 3-width visual verification)
   apply to generated code exactly as to handwritten code.
3. **Disclose substantial AI authorship** in the PR description — it helps
   review calibration, and it's honest.
4. **No drive-by mass rewrites.** Large mechanical diffs (reformat-the-world,
   rename sweeps) generated by an agent without prior discussion will be
   closed.

## Hard product rules

- **No auto-submit features, ever.** PRs that automate the final
  Submit/Send/Apply step of a job application are rejected on principle.
- **User data stays local and gitignored.** Nothing under `inputs/`, `data/`,
  or `artifacts/` may become tracked, and no feature may ship it off-machine
  beyond the user's own AI provider calls.
- **No telemetry.** Sur9e collects nothing.
- **English only** in code, comments, and docs.

## Commit & PR style

- Conventional-ish prefixes: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`
  — scope in parentheses when useful (`fix(table): …`).
- Logically grouped commits; one concern per PR.
- Frontend changes: include desktop (1280×800), tablet (768×1024), and mobile
  (375×667) screenshots in the PR.

## Maintainer-only tools

A few scripts exist for repo maintenance and are not part of the user
workflow: `cli/refresh-openrouter-pricing.mjs` (refresh bundled pricing
data) and the idempotent `scripts/migrate-*.mjs` data migrations. You can
ignore them unless a change touches what they maintain.

MCP servers ship pre-wired: the tracked `.mcp.json` (Claude), `opencode.json`
(OpenCode), and `.codex/config.toml` (Codex) register a stdio Playwright MCP
for browser-rendered job-description extraction and offer verification — no
setup step. Web search and fetch use each CLI's native tools.
